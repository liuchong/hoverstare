# 01 — Spec 优先的开发纪律

1. **`specs/` 是单一事实来源**。任何功能开发必须先有对应 spec；
   实现与 spec 不一致时，以 spec 为准——或者**先改 spec**。
2. spec 不是金科玉律：发现不合理就改 spec，但**禁止代码直接偏离 spec**
   （先改文档，再改代码，顺序不可反）。
3. 每个 spec 对应 `src/` 下同名模块；新增模块先补 spec。
4. 里程碑的验收记录直接写在 `specs/README.md` 对应小节里，
   实测发现的经验教训也记录在案（供后续里程碑参考）。
5. 重大真实环境验证结果写入 `specs/validation-<date>.md`。
