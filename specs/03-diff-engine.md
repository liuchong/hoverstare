# 03 — Diff 引擎

## 目标

把 unified diff 解析成结构化数据，回答两个核心问题：

1. **审查范围**：哪些文件、哪些行被改了（喂给模型）；
2. **可评论行映射**：每个文件 RIGHT 侧哪些行号可以挂行内评论（发布前校验用）。

## 接口

```rust
pub struct ParsedDiff { pub files: Vec<FileDiff> }

pub struct FileDiff {
    pub path: String,                // 新路径（rename 后为 b/ 侧）
    pub old_path: Option<String>,    // rename 时的旧路径
    pub kind: FileKind,              // Added | Modified | Deleted | Renamed
    pub hunks: Vec<Hunk>,
    /// RIGHT 侧所有可评论行号（context + added 行）
    commentable: BTreeSet<u64>,
}

pub struct Hunk {
    pub old_start: u64, pub new_start: u64,
    pub lines: Vec<DiffLine>,        // Context / Added / Deleted
}

impl ParsedDiff {
    pub fn parse(input: &str) -> ParsedDiff;             // 容错解析，不返回 Err
    pub fn commentable_lines(&self, path: &str) -> Option<&BTreeSet<u64>>;
    /// 找到 path 的 line 最近的合法锚点（先同 hunk 最近 added/context 行，再全局最近行）
    pub fn nearest_anchor(&self, path: &str, line: u64) -> Option<u64>;
    pub fn added_line_content(&self, path: &str, line: u64) -> Option<&str>; // 指纹用
}
```

## 解析规则（状态机）

两个状态：**文件头** / **hunk 体**。

- `diff --git a/X b/Y`：新文件开始 → 进入文件头状态，重置当前文件；
- 文件头状态中只有 `+++ b/P` 设定当前文件路径（`+++ /dev/null` 表示删除文件）；
  注意：hunk 体内以 `+++ b/` 开头的**内容行**（比如有人往文档里贴了一段 diff）
  不得误判为文件头——只有文件头状态才识别 `+++`；
- `@@ -a[,b] +c[,d] @@`：进入 hunk 体，`new_line = c`；
- hunk 体内：
  - ` ` 前缀（context）：记入 commentable，`new_line += 1`；
  - `+` 前缀（added）：记入 commentable，`new_line += 1`；
  - `-` 前缀（deleted）：只记录，`new_line` 不变；
  - `\ No newline at end of file`：忽略；
- 解析必须**容错**：遇到不认识的行跳过继续，永不 panic、永不返回 Err
  （输入是外部数据 + 模型可见数据，必须视为不可信）。

## 过滤（在解析后、喂模型前）

两层过滤，按顺序执行：

1. **内置规则**（不可配置关闭，可配置追加）：
   - 常见锁文件：`Cargo.lock`、`package-lock.json`、`pnpm-lock.yaml`、`yarn.lock`、`poetry.lock`、`go.sum`、`composer.lock`、`Gemfile.lock`
   - 压缩产物：`*.min.js`、`*.min.css`、`*.map`
   - 含 `Code generated by ... DO NOT EDIT` 头标记的文件（检查 patch 前 5 行新增内容）
   - `.github/` 目录（CI 配置不是应用代码，且降低 prompt injection 面）
2. **用户配置**：`config.ignore` 的 glob 规则（`globset` 匹配，相对仓库根路径）。

被过滤的文件不进 prompt、不进可评论集合，但在摘要元数据里记录 `excluded_files` 数量。

## 截断（预算控制）

若过滤后 diff 总大小 > `max_diff_kb`：

1. 文件按优先级排序：源代码（`*.rs/go/py/js/ts/java/...`）> 测试 > 文档/配置 > 其他；
2. 依序保留，整文件粒度截断（不截半个文件）；
3. 被截断的文件列表以 `TRUNCATED FILES` 段落附在 prompt 末尾告知模型；
4. 截断后仍超预算 2 倍 → 分析跳过（exit 0，输出 warning）。

## 测试要点（fixture 驱动，`tests/fixtures/diff/`）

- 普通多文件多 hunk；新增文件；删除文件（commentable 为空）；
- rename（`rename from/to`），`old_path` 正确；
- hunk 内容里含字面 `+++ b/foo`、`diff --git`、`@@` 的行；
- `\ No newline at end of file`；
- 空 diff；只有删除的 diff；
- files API 重组的 diff（spec 02）与原生命令输出解析结果一致；
- `nearest_anchor`：行本身合法 / 同 hunk 吸附 / 跨 hunk 落到全局最近行 / 文件不存在 → None。
